# Security policy

MurOS is a firewall. A vulnerability in MurOS can expose the network
it protects, so we take reports seriously.

## Supported versions

Only the latest stable release and the current release candidate
receive security fixes. Older releases are not patched. The supported
versions are listed on the GitHub releases page.

## Reporting a vulnerability

Please do not open a public GitHub issue for security problems.

Report privately through one of these channels:

- GitHub private vulnerability reporting:
  <https://github.com/murosorg/muros/security/advisories/new>
- Email: security@muros.org (PGP key published on the project site
  when available)

Include:

- A description of the issue and its impact
- Steps to reproduce, ideally with a minimal proof of concept
- The MurOS version and the Debian version where you reproduced it
- Your name or handle for the credit line, or a note if you want to
  stay anonymous

## What to expect

- We acknowledge the report within 5 business days.
- We confirm or reject the issue within 15 business days.
- For confirmed issues we agree on a disclosure timeline with the
  reporter. The default is 90 days from acknowledgement, shorter if
  the issue is actively exploited.
- A fixed release is published with a CHANGELOG entry, a GHSA advisory
  and, when applicable, a CVE.
- Reporters are credited unless they ask not to be.

## Scope

In scope:

- The MurOS backend, frontend and packaging scripts in this repository
- The default install produced by `install.sh` on a fresh Debian 13
- The configuration generated by MurOS for nftables, nginx, keepalived,
  WireGuard, strongSwan, dnsmasq, unbound, fail2ban, snmpd

Out of scope:

- Vulnerabilities in Debian packages MurOS depends on. Report those to
  the Debian security team. We will track them and bump versions when
  patched packages are available.
- Issues that require a pre-existing root shell on the box.
- Self-inflicted misconfigurations (for example: opening the admin UI
  on a WAN interface and using a weak password).
- Denial of service through resource exhaustion when the attacker has
  legitimate API credentials.

## Hardening defaults

MurOS ships with a number of hardening defaults: HTTPS-only admin UI,
fail2ban on the API and SSH, TOTP MFA available for the admin account,
password policy with anti-leak checks, JWT tokens limited to 8 hours,
audit log of every UI action. The default install does not expose any
service on the WAN interface beyond what the user explicitly enables.

If you find a default that you consider unsafe, that is a valid
security report even if it is not technically a vulnerability.
